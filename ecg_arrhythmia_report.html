Executive Summary

Short version: the project became credible only after the evaluation protocol got stricter and more balanced. Model improvements mattered, but evaluation fixes mattered first.

Most important positive result

Once the pipeline moved from optimistic random record splits to strict de Chazal inter-patient testing, the strongest generic model became rr-context. It can beat the poster target on a single strict run and it now has a full 5-fold x 3-seed mean, which still comes out ahead of the older generic models.

Most important caveat

The best personalized model is stronger on fold 0, but it is a different evaluation setting because it uses causal within-record history. It also costs more than twice the CPU latency of rr-context, so it should not be sold as a free benchmark gain.

Reading Guide

Generic base. The model already uses multiscale morphology, short-range rhythm context, explicit RR features, and an RR-normalized beat view before any personalization is added.

Personalized add-on. The history branch only sees earlier beats from the same record. It never uses future beats.

Why this layout matters. The first past-only version collapsed because it threw away too much strong generic context. The working version keeps the generic trunk and lets personalization act as calibration.

  • Fast edge option: use rr-context only.
  • Higher-accuracy personalized option: add the causal history branch.
  • Current ablation question: within that history branch, is the RR baseline more useful, the history morphology prototype more useful, or do they need to be combined?

Problems, Solutions, and What Each Fix Taught Us

The project improved by tightening the science as much as by changing the architecture.

1. Random split made the prototype look finished

Problem. Random record splits produced excellent test scores, but those scores were not stable enough to support a paper claim.

Solution. Switched to the classic de Chazal inter-patient split and added multi-seed sweeps.

Lesson. The first big drop in score was not the code getting worse. It was the evaluation getting more honest.

2. DS1 fold construction was distorting validation

Problem. DS1 ventricular beats were highly concentrated in a few records like 208, 223, 119, and 203.

Solution. Replaced record-level stratification with beat-balanced fold assignment.

Lesson. Validation engineering mattered enough to move whole conclusions.

3. Single-view morphology was not enough

Problem. A single beat does not fully capture ventricular timing irregularities or short-term rhythm context.

Solution. Added multiscale morphology and then a local context branch.

Lesson. ECG detection is partly a rhythm problem, not just a shape problem.

4. Heart-rate variation across patients changed the time scale

Problem. The same pathology can be compressed or stretched depending on patient heart rate.

Solution. Added explicit RR features and a beat view resampled to the local cardiac cycle.

Lesson. Scale adaptation was real signal, not just feature decoration.

5. Naive personalization collapsed

Problem. The first past-only personalized model threw away too much strong local context and failed badly on the strict test split.

Solution. Kept rr-context as the base and added causal history as an auxiliary branch instead of replacing the generic branch.

Lesson. Personalization helps most when it nudges a strong generic model instead of trying to replace it.

6. Better scores can still be the wrong deployment choice

Problem. The strongest personalized numbers also came with a large latency increase.

Solution. Benchmarked CPU latency after each major model jump and postponed quantization work until the full-precision comparison became clear.

Lesson. For edge work, a metric gain is only half the story.

Algorithm and Evaluation Techniques Used

This is the complete toolbox used across the project, including tricks that were kept, tricks that were retired, and tricks that mainly improved evaluation honesty.

SE blocks plus temporal attention baseline The early attention model used squeeze-excitation and temporal self-attention. It helped on easy splits, but it was not the whole answer under strict testing.
Multiscale morphology encoder Parallel receptive fields at different time scales made the model less brittle to patient-to-patient rate and waveform variation.
Short-range rhythm context A small GRU over nearby beats let the model judge a beat inside its local rhythm instead of as a completely isolated waveform.
Explicit RR features The RR-aware generic model used beat timing features directly instead of hoping the morphology branch would infer them implicitly.
RR-normalized beat view Resampling the center beat to the local cardiac cycle gave the model a more rate-invariant morphology view.
Causal history morphology prototype The personalized branch summarizes earlier beats from the same record into a morphology prototype and compares it to the current beat.
Rolling RR baseline and z-score calibration The personalized branch also keeps per-record RR mean and standard deviation so the current beat can be judged relative to recent personal timing.
R-peak aligned fixed windows A simple preprocessing choice that kept the main morphology in a stable location before any learned modeling happened.
Patient-wise de Chazal inter-patient split This was the core evaluation setting for unseen-patient generalization and replaced the earlier optimistic random record split.
Beat-balanced DS1 validation folds The biggest evaluation fix: balance ventricular-beat counts across validation folds, not just record labels.
Weighted sampler without extra class-weighted loss This became a better default than stacking both imbalance tricks at once.
Validation threshold tuning Operating-point selection mattered, especially whenever precision and recall were trading against each other.
Multi-seed sweeps and cross-fold testing Many settings looked stronger under one seed or one fold than they did once the comparison was widened.
CPU latency benchmarking after each major jump This kept the report tied to edge deployment reality instead of only reporting accuracy gains.
TorchScript and dynamic quantization export hooks Even before serious compression work, the codebase was prepared to test deployment artifacts instead of training-only checkpoints.

Why the Results Look the Way They Do

The numbers are easier to understand when grouped by mechanism instead of by file.

Why the early random split looked unrealistically strong

The model was not necessarily learning the full unseen-patient problem. It was benefiting from an easier split where train and test records were more alike.

That is why both attention and baseline could post spectacular F1 scores before the methodology was tightened.

Why beat-balanced validation mattered so much

If one validation fold contains the high-ventricular records and another does not, checkpoint selection becomes noisy and misleading.

Balancing ventricular-beat counts made the fold means far more interpretable.

Why RR-aware models improved both precision and recall

ECG abnormality is partly about timing, not just local morphology. Explicit RR features and a normalized beat view gave the model a better way to compare patients with different heart rates.

Why personalization helps, but needs a label

The causal history branch gives the model a personal baseline for the current record, which can reduce false positives and still preserve strong recall.

The full 15-run cross-fold sweep says the gain is real but smaller than the first seed-42-only scan suggested. It is still a personalized setting, so it answers a different question from pure generic unseen-patient classification.

What to Say in One Paragraph

A compact version you can reuse when sending an update.

The prototype started with over-optimistic random-split results, then became much more believable after switching to strict de Chazal testing and beat-balanced validation folds. Multiscale morphology plus rhythm context helped, RR-aware inputs helped more, and the best generic strict single run reached P 0.9166 / R 0.9689 / F1 0.9420. A personalized causal-history branch still looks promising too: in the full 5-fold x 3-seed sweep, its mean reached P 0.8177 / R 0.9676 / F1 0.8817, with a best single run of P 0.9265 / R 0.9783 / F1 0.9517, but at roughly double the latency and under a different personalized setting. A focused ablation on fold 0 suggests most of that personalized gain comes from the history morphology prototype, while the RR baseline adds a smaller extra boost.
